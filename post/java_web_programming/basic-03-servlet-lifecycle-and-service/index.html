Servlet 라이프 사이클


Servlet 라이플 사이클 [출처] slipp.net

WAS는 서블릿 요청을 받으면(=URL로 클라이언트가 서버에 요청하면) 해당 서블릿이 메모리에 있는지 확인한다.

만약 메모리에 없으면 해당 서블릿 클래스를 메모리에 올리고(생성자 실행), init(), service()를 실행한다. 또한 WAS는 HttpSevletRequest, HttpServletResponse 객체를 생성해서 service() 메서드의 파라미터로 전달한다. 따라서 요청에 대한 처리, 응답은 service() 메서드를 오버라이딩해서 구현하는 것이다.

만약 메모리에 클라이언트가 요청한 서블릿이 있다면 서블릿 객체를 생성하지 않고 service()만 호출한다.

WAS가 종료되거나, 웹 애플리케이션이 새롭게 갱신될 경우 메모리에 올라간 서블릿 객체는 더 이상 사용될 수 없다. 따라서 destroy()가 실행된다


service()와 템플릿 메서드 패턴

service()는 템플릿 메서드 패턴으로 구현되어 있다

템플릿 메서드 패턴( Template Method Pattern )

상위 클래스를 상속 받은 자식 클래스들의 전체적인 레이아웃을 통일 시키면서, 어느정도 유연성을 주도록하는 디자인 패턴이다. 통일을 위해서 하위 클래스에서 수정하면 안되는 부분은 final로, 하위 클래스 모두 공통된 기능을 가지지만 그 세부 내용이 다른 경우에는 abstract로 상위 클래스에 정의한다. 또한 유연성을 제공하기 위해 훅 메서드(hook method)를 정의해서 상속 받은 클래스가 선택적으로 오버라이딩 할 수 있도록 한다. [ 참고 ]

Servlet 라이프 사이클에 따르면 service()가 호출되어야 하는데, Servlet에 doGet(), doPost() 등 다른 메서드만 구현해도 동작한다. 왜냐하면, service()가 템플릿 메서드 패턴으로 구현되었기 때문에 Servlet 내에 service()가 없으면 부모인 HttpServlet의 service()가 실행된다. 그리고 HttpServlet을 상속받은 Servlet은 선택적으로 doXXX()를 오버라이딩 할 수 있기 때문이다.
