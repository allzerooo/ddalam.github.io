컨테이너

인스턴스의 라이프 사이클을 관리, 생성된 인스턴스에게 추가적인 기능을 제공

예를 들어, 개발자가 Servlet을 인스턴스화 하는 과정을 거치지 않아도 Tomcat(WAS)이 가지고 있는 Servlet 컨테이너가 Servlet을 메모리에 올린 후 실행한다


IoC ( Inversion of Control )

의존성에 대한 제어권이 역전되었다

의존성

class Engine() {

}

class Car() {
  Engine engine;

  public Car() {
    this.engine = new Engine();
  }
}

자동차와 엔진의 관계에서(자동차가 엔진을 생성) 자동차가 엔진에 의존한다라고 할 수 있다


의존성을 외부로

class Engine() {

}

class Car() {
  Engine engine;

  public Car(Engine engine) {
    this.engine = engine;
  }
}
Engine engine = new Engine();
Car car = new Car(engine);

엔진에 대한 의존성을 자동차 자체적으로 해결하지 않고 자동차 객체 외부에서 생성된 엔진 객체를 자동차 객체의 생성자 인자로 주입하는 형태


의존성 제어권의 역전

의존하는 클래스가 직접 의존성을 만들지 않고, 누군가 만들어준 의존성이 타입만 맞으면 어떤 것이든 상관없이 받기만 하면 정상 작동되도록 하는 것이다.



IoC 컨테이너

Spring IoC 컨테이너는 ApplicationContext와 BeanFactory가 있는데, ApplicationContext는 BeanFactory를 상속받을 뿐 아니라 더 다양한 기능을 제공한다.

IoC 컨테이너는 빈(bean)을 만들고, 빈들 사이에 의존성을 얶어주고, 의존성을 제공하는 일을 한다. bean은 Spring IoC 컨테이너가 관리하는 객체이다. 모든 객체가 빈으로 등록되는 것은 아니다. IoC 컨테이너가 의존성을 관리, 주입해주는 대상은 빈으로 등록된 객체들에만 해당된다. ( 빈으로 등록하는 방법은 여러가지가 있다 )

ApplicationContext에서 직접 꺼낸 빈과 ApplicationContext가 자동으로 주입해준 빈을 비교해보면 같은 인스턴스이다. 이런 객체를 싱글톤 스코프 객체라고 하는데, 사용될 때마다 매번 생성되는 것이 아니라 애플리케이션 전반에서 계속 재사용할 수 있는 객체를 말한다. 이 또한 IoC 컨테이너가 관리해준다.
